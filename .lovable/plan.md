## القسم 1: الأمان والبنية التحتية (الأسبوع 1-2)

هذا يجب أن يكون **أول شيء** لأن كل التطويرات الأخرى تعتمد عليه.

### 1.1 إصلاح RLS الحرجة (6 مشاكل)
- تقييد `employees` (national_id, bank_account) لـ HR والموظف نفسه
- تقييد `suppliers` (IBAN, bank_account) للمحاسبين فقط
- إضافة Views آمنة لـ `customers` تخفي البيانات الحساسة عن غير المخولين
- تعزيز عزل `profiles` ضد timing attacks
- تشفير `user_2fa_settings` (مفاتيح TOTP) باستخدام pgcrypto
- إضافة طبقة حماية إضافية لـ `bank_accounts`

### 1.2 إصلاح RLS التحذيرية (7 مشاكل)
- تقييد INSERT على `activity_logs` للـ triggers فقط
- إضافة RLS لـ `suspicious_activities` و `security_dashboard`
- تقييد `user_login_history` للمستخدم نفسه والأدمن

### 1.3 جودة الكود
- استبدال `error.message` بـ `getSafeErrorMessage()` في 10 ملفات
- استبدال `any` بأنواع TypeScript محددة في الملفات الأساسية (InvoicesPage, AdminDashboard, RolesPage, إلخ)

## القسم 2: الفواتير والمبيعات (الأسبوع 3-4)

أكثر الأقسام استخداماً - تحسين شامل.

### 2.1 واجهة المستخدم
- تقسيم `InvoiceFormDialog.tsx` (533 سطر) إلى مكونات أصغر
- تحسين تجربة الموبايل لصفحة الفواتير (التمرير، الأزرار، المساحات)
- إضافة Skeleton loaders أسرع ومخصصة لكل حالة
- تحسين نموذج الفاتورة: autocomplete للعملاء والمنتجات أسرع

### 2.2 الأداء
- تفعيل Virtual Scrolling فعلياً للقوائم الطويلة (الكود موجود لكن يحتاج ضبط)
- تحسين استعلامات الفواتير مع pagination من الخادم بدل تحميل الكل
- إضافة debounce للبحث في الفواتير

### 2.3 وظائف جديدة
- نظام نسخ الفاتورة (Duplicate Invoice)
- فاتورة مرتجع (Credit Note)
- طباعة دفعية (طباعة عدة فواتير دفعة واحدة)
- إضافة ملاحظات داخلية على الفاتورة (غير مرئية للعميل)
- تحسين تدفق: عرض أسعار → أمر بيع → فاتورة (بضغطة واحدة)

### 2.4 التكامل
- ربط الفاتورة بحركة المخزون تلقائياً عند التأكيد
- إشعار تلقائي عند اقتراب موعد استحقاق الفاتورة
